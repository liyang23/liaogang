## 一、问题陈述

> 辽港伐谋项目致力于港口运筹优化算法的研发与应用。核心工作流程为：业务专家与算法工程师将港口运筹文档中的知识编码为约束、规则、参数等结构化要素，再将要素装配成提示词，输入伐谋大模型以生成优化方案。

> 目前团队面临以下痛点：
>
> 1. **知识散落不可追溯**：港口运筹知识分散在项目需求说明书、码头反馈记录、算法提示词等不同文档中，缺乏统一的结构化存储与检索机制。根据项目现状统计，约 **70 个文档**（PDF、Word、Excel）中的知识未被系统化管理。
>
> 2. **知识冲突不可见**：当不同来源的知识对象存在值冲突、作用域重叠、语义矛盾时，缺乏自动检测与治理机制。当前活跃知识对象中存在至少 **12 个对象间冲突**和 **1 个自身体健康问题**，可能直接影响伐谋大模型输出的正确性。
>
> 3. **提示词装配效率低下**：从知识对象到最终提示词，依赖人工复制粘贴与手动校对，每次提示词版本迭代平均耗时约 **2-3 小时**。当前统筹模型主提示词 V3.0 包含 **113 个知识对象**，字符数约 4,821。
>
> 4. **版本变更缺乏追溯**：知识对象的数值变更（如参数调优）缺乏与提示词版本的关联追溯，算法输出结果无法回溯到具体的提示词装配方案。
>
> 5. **知识对象缺乏生命周期管理**：从创建、审核、发布、使用到废弃缺乏标准化工作流。当前 **278 个知识对象**中存在 8 个草稿、17 个待审核、14 个已废弃但未清理。
>
> 6. **知识资产缺乏项目级隔离**：所有 278 个知识对象处于扁平命名空间，分属不同码头、不同优化场景的知识混在一起，无法按项目进行数据资产隔离与访问控制。当前无项目概念，所有用户对全部 KO 可见，无法满足不同业务组独立管理知识资产的需求。

> 如果不解决，将导致算法提示词质量不可控、团队协作效率持续低下、知识资产无法复用，最终影响伐谋大模型的优化方案质量与港口运营决策的可靠性。

### 5.2 状态机与横向约束

> 本节集中定义跨模块共享的状态机、角色映射、算法规则和审计要求，避免在各 FR 中重复说明。

#### 5.2.1 KO 状态机（v0.33 双层重构）

##### 5.2.1.1 双层状态定义

| 层级 | 实体 | 状态 | 含义 |
|------|------|------|------|
| **L1：KO 主体状态** | `ko.status` | `Active` | 当前生效版本，已被提示词装配引用 |
| | | `Deprecated` | 已废弃，仅保留历史 |
| | | `Pending` | 有工作版本正在 Review 期间，主体 KO 仍为旧版本 |
| **L2：KO 工作版本状态** | `ko_version.status` | `Draft` | 业务专家在编辑中 |
| | | `Review` | 已提交审核，等待审核员 |
| | | `Approved` | 审核通过，待发布 |
| | | `Rejected` | 审核驳回，可继续编辑或撤回 |
| | | `Withdrawn` | 提交者主动撤回 |
| | | `Published` | 已发布为 KO 的新 Active 版本 |
| | | `Discarded` | 已废弃（不发布）|

> **关键设计**：
> - KO 主体只有 3 个状态（Active / Deprecated / Pending）
> - KO 工作版本（草稿/审核中的版本）独立维护状态
> - 业务专家修改 Active KO 不会改变 KO 主体状态，而是创建一个新的 KO Version 走工作流
> - KO 主体进入 `Pending` 状态当且仅当存在 `Review/Approved` 状态的 KO Version

##### 5.2.1.2 状态转换图（双层）

```
[KO 主体]
[新建 KO] → Active (类型豁免：DOC/PRM 流程见 §5.2.1.4)
[Active]  → Pending (有 in-flight KO Version 进入 Review)
[Pending] → Active (KO Version Published，替换原 Active)
[Active]  → Deprecated (标记废弃)
[Deprecated] → Active (管理员恢复，且无 in-flight Pending)

[KO 工作版本]
[无]  → Draft (业务专家在 Active KO 上点击"编辑"或新建 KO 时创建)
[Draft] → Draft (编辑保存，PATCH/MINOR 自增)
[Draft] → Review (提交审核)
[Draft] → Discarded (创建者删除草稿)
[Review] → Approved (审核员通过，单次审核，v0.38 删除 AB 岗)
[Review] → Rejected (审核员驳回，附理由)
[Review] → Withdrawn (提交者主动撤回)
[Approved] → Published (自动或手动发布，替换 KO 主体 Active 版本)
[Approved] → Withdrawn (Approved 状态下提交者撤回)
[Rejected] → Draft (继续编辑)
[Rejected] → Discarded (放弃)
[Withdrawn] → Draft (继续编辑)
[Withdrawn] → Discarded (放弃)
```

##### 5.2.1.3 状态转换触发条件与角色限制

| 转换 | 触发动作 | 角色限制 | 附加条件 |
|------|----------|----------|----------|
| **KO 主体** ||||
| 新建 KO → Active | 创建成功（DOC/PRM 走豁免流程）| 业务专家 / 算法工程师 | 类型豁免见 §5.2.1.4 |
| Active → Pending | 有 KO Version 进入 Review | 系统自动 | 不能同时存在多个 Review 状态的 KO Version |
| Pending → Active | KO Version Published | 系统自动 / 管理员手动 | 原 Active 版本归档为 v(N-1) 历史 |
| Active → Deprecated | 标记废弃 | 创建者 / 管理员 | 提示词装配时该 KO 不再可选 |
| Deprecated → Active | 管理员恢复 | 系统管理员 | 仅当无 Pending 状态 KO Version |
| **KO 工作版本** ||||
| 无 → Draft | ① 新建 KO ② 在 Active KO 上"编辑" | 业务专家 / 算法工程师 | 编辑现有 KO 时需校验编辑权 |
| Draft → Review | 提交审核 | 创建者 | 校验必填字段完整性 |
| Review → Approved | 审核通过 | 审核员 | 单次审核（v0.38 删除 AB 岗） |
| Review → Rejected | 审核驳回 | 审核员 | 必填理由（≥10 字符）|
| Review → Withdrawn | 撤回审核中版本 | 提交者 | - |
| Approved → Published | 发布 | 管理员 / 系统自动 | 校验无未处置冲突 |
| Rejected → Draft | 继续编辑 | 提交者 | - |
| Rejected → Discarded | 放弃 | 提交者 | - |
| Withdrawn → Draft | 继续编辑 | 提交者 | - |
| Withdrawn → Discarded | 放弃 | 提交者 | - |
| Draft → Discarded | 删除草稿 | 创建者 | 写审计（见 §5.2.6）|

##### 5.2.1.4 类型豁免规则（v0.33 新增）

> §5.2.1 状态机的默认流程适用于 **CON / RUL / PAR / SCH** 四种类型；**DOC / PRM 类型有特殊豁免**。

| 类型 | 豁免规则 | 状态机变体 |
|------|----------|------------|
| **DOC** | 上传后直接为 Active（不经 Review）| 新建 → Active 直跳（无 in-flight）|
| **DOC** | 编辑时进入 in-flight Draft，但发布仍需审核（与 CON 一致）| 编辑沿用 §5.2.1.2 工作版本流程 |
| **DOC** | 待审核 Tab 在 FR-39 不显示（仅显示 Active / Deprecated）| UI 层面豁免 |
| **PRM** | 新建 → Draft（无 in-flight 概念，PRM 整体审核）| 新建走 §5.2.1.2 工作版本流程 |
| **PRM** | PRM 编辑产生新 version（不替换原 PRM）| 特殊：PRM 的"修改"是发新版本（v2.0.0），不覆盖原版 |
| **PRM** | Section 结构变更触发 MAJOR 升级 | - |
| **CON/RUL/PAR/SCH** | 默认完整流程 | 走 §5.2.1.2 |

> **说明**：DOC"上传即 Active"的语义是"DOC 内容已由业务专家签字确认（上传动作即确认）"，因此豁免审核。但 DOC"编辑"仍需审核（编辑可能引入错误信息）。

##### 5.2.1.5 约束规则（v0.33 强化，v0.38 重编号）

- 任何状态转换必须记录审计日志（操作人 / 源状态 / 目标状态 / 理由）
- Review 状态 KO Version 不参与提示词装配（不计入 Composer KO 桶）
- Deprecated 状态 KO 不出现在 KO 库默认搜索结果，可通过"显示已废弃"开关查看
- **不允许从 Active KO 直接创建新的 Active 版本的 KO**（必须经过 Draft → Review → Approved → Published 完整流程）
- **Pending 状态 KO 期间，不允许创建新的工作版本**（避免版本冲突）
- **审核员不可审核自己提交的 KO Version**（即使角色是审核员，也不能审自己）
- **同一 KO 同时最多存在 1 个 in-flight 工作版本**（避免多版本竞争）

#### 5.2.2 系统角色权限模型（v0.37 简化重写）

#### 5.2.2.1 业务专家提案权流程（v0.34 新增，v0.37 简化）

**核心设计**：
- 业务专家对 CON / SCH / DOC **无直接编辑权**（默认）
- 但有 **提案权**（创建 `ko_proposal` 记录，描述修改建议）
- 算法工程师或管理员审核后决定是否采纳
- 提案被采纳后，**修改由审核人执行**（而非业务专家），业务专家获得"被采纳"通知

**数据模型**：

```sql
-- 新增 ko_proposal 表
CREATE TABLE ko_proposal (
  id            VARCHAR(30) PRIMARY KEY,  -- PROP-{YYYYMMDD}-{NNNNNN}
  ko_id         VARCHAR(20) NOT NULL,    -- KO-CON-XXXX（被提案的 KO；CREATE 类型时为 null）
```

#### 5.2.3 冲突特征指纹算法

**目的**：在 KO 内容变更、重新检测时，能稳定识别"同一冲突"并沿用旧 ID。

**指纹函数**：

```python
import hashlib

def conflict_fingerprint(ko_a_id: str, ko_b_id: str, conflict_type: str,
                          scope_key: str, field_key: str) -> str:
    """
    冲突指纹：MD5(ko_a_id + ko_b_id + conflict_type + scope_key + field_key)
    - ko_a_id, ko_b_id 按字典序排序（保证双方视角同一指纹）
    - conflict_type ∈ {C1, C2, C3, C4, C5, C6, H1, H2, H3, H4, H5, H6}
    - scope_key: 作用域标识（如 "码头=大窑湾,设备=岸桥"），跨作用域冲突用 ""
    - field_key: 冲突字段标识（如 "value", "hardness", "dependencies"）
    """
    pair = sorted([ko_a_id, ko_b_id])
    raw = f"{pair[0]}|{pair[1]}|{conflict_type}|{scope_key}|{field_key}"
    return hashlib.md5(raw.encode()).hexdigest()[:12]
```

**复用规则**：

| 情形 | 处理 |
|------|------|
| 新检测 + 指纹未命中 | 创建新冲突 ID（CT-{TYPE}-{YYYYMMDD}-{NNNN}） |
| 新检测 + 指纹命中 + 状态=open | 沿用旧 ID，**更新 last_detected_at** |
| 新检测 + 指纹命中 + 状态=resolved | 重开（沿用 ID，状态变回 open，写历史快照） |
| KO 内容变更后冲突不再成立 | 标记 `auto_resolved: true`（不删除记录，保留历史） |

**FR-21 / FR-22 验收补**：
- 冲突 ID 沿用基于指纹算法；不依赖 KO ID 排序的偶然一致
- `last_detected_at` 用于冲突"复活"判定
- `auto_resolved` 标记的冲突在治理页"已处置"组显示，但标注"自动消失"灰色字

#### 5.2.4 SNP 与 PRP 关系（v0.33 重构）

**核心原则**：
- SNP 反映"装配方案的版本变化"（模板结构 + KO 引用 + KO 当前值）
- PRP 反映"渲染动作"（用户操作 + 渲染参数 + 触发时间）
- hash 只包含"SNP 维度"字段，不含"PRP 维度"字段

**数据模型**：

```
┌──────────────────┐         ┌──────────────────┐
│   PRM（模板）     │         │   KO（Active）    │
│   KO-PRM-XXXX    │         │   KO-CON-XXXX     │
│   v{MAJOR.MINOR.PATCH}      │   v{MAJOR.MINOR.PATCH}
└────────┬─────────┘         └────────┬─────────┘
         │                            │
         │  引用                      │ 装配
         ↓                            ↓
┌──────────────────────────────────────────┐
│  SNP 候选 = PRM + KO 集合 + KO 字段值       │
│  ko_assembly_hash = SHA256(SNP 候选)        │
│  → 命中 → 复用 SNP → 仅创建 PRP            │
│  → 未命中 → 创建新 SNP + 第 1 个 PRP        │
└────────┬─────────────────────────────────┘
         │
         ↓
┌──────────────────┐         ┌──────────────────┐
│   SNP（快照）     │ 1     N │   PRP（渲染记录）  │
│   SNP-YYYYMMDD-HHMM       │   PRP-{NNNN}    │
│   ko_assembly_hash        │   snapshot_id   │
│   rendered_text_canonical │   rendered_text │
│   stale: bool             │   force_rendered│
│                           │   render_time   │
│                           │   token_count   │
│                           │   user_id       │
│                           │   reason        │
└──────────────────┘         └──────────────────┘
```

**hash 输入字段定义**（v0.33 新增）：

| 字段 | 来源 | 是否入 hash | 理由 |
|------|------|------------|------|
| `prm_id` | PRM 模板 ID | ✅ | 模板结构变更需新 SNP |
| `prm_version` | PRM 模板版本号 | ✅ | 模板版本变更需新 SNP |
| `ko_ids[]` | 装配 KO 列表（排序后）| ✅ | KO 集合变化需新 SNP |
| `ko_versions[]` | 装配 KO 的版本号列表 | ✅ | KO 版本升级需新 SNP |
| `ko_field_values[]` | KO 当前值（如 PAR 值）| ✅ | KO 数值变更需新 SNP |
| `manual_subitems[]` | 手动录入子项的 hash | ✅ | 手动子项内容变更需新 SNP |
| `var_bindings[]` | 变量绑定关系（var → PAR 引用）| ✅ | 绑定关系变更需新 SNP |
| `user_id` | 渲染用户 | ❌ | 同一装配不同用户应复用 SNP |
| `rendered_text` | 渲染后文本 | ❌ | PRP 字段 |
| `render_time` | 渲染耗时 | ❌ | PRP 字段 |
| `token_count` | Token 数 | ❌ | PRP 字段 |
| `force_rendered` | 是否强制渲染 | ❌ | PRP 字段 |
| `reason` | 强制渲染理由 | ❌ | PRP 字段 |

**hash 计算**：

```python
def ko_assembly_hash(snp):
    fields = [
        snp['prm_id'],
        snp['prm_version'],
        sorted(snp['ko_ids']),
        sorted(snp['ko_versions']),
        sorted(snp['ko_field_values']),
        sha256(json.dumps(snp['manual_subitems'], sort_keys=True)),
        sorted(snp['var_bindings']),
    ]
    raw = '|'.join(str(f) for f in fields)
    return hashlib.sha256(raw.encode()).hexdigest()[:16]
```

**生成规则**（v0.33 修订）：

1. 每次点击"渲染提示词"：
   - 计算 `snp_candidate`（SNP 维度字段）
   - 计算 `ko_assembly_hash = SHA256(snp_candidate)`（截 16 位）
2. 若 hash ≠ 任何现有 SNP 的 hash → **创建新 SNP** + 第 1 个 PRP
3. 若 hash 已存在 → **仅创建新 PRP**，`snapshot_id` 关联到已有 SNP
4. PRP 字段（rendered_text、render_time、token_count、force_rendered、reason、user_id）独立存储
5. SNP 反映"装配方案的版本变化"，PRP 反映"渲染动作"

#### 5.2.5 字典引用完整性

字典项（效力/权威/类型分组/量纲/类型介绍）可被 KO 引用，必须保证引用完整性。

**三种处置策略**：

| 策略 | 适用 | 行为 |
|------|------|------|
| **软删除**（推荐） | 效力 / 权威 / 类型分组 / 量纲 | 标记 `disabled: true`；存量 KO 仍显示该字典项但带 `ⓘ 已停用` 灰色标签；KO 创建/编辑表单中**不可选**该字典项 |
| **硬删除** | 仅未被引用的字典项 | 删除前校验 `SELECT COUNT(*) FROM ko WHERE ... LIKE '%"code":"XXX"%'`，无引用才允许 |
| **级联更新** | 字典项重命名（不改 code） | 直接更新名称，旧名称记录审计 |

**FR-29 / FR-35 验收补**：
- 删除字典项前必须通过引用校验（命中引用 → 提示"被 N 个 KO 引用，请改用软删除"）
- 软删除字典项在 KO 创建/编辑表单下拉选项中不出现
- 存量 KO 引用软删除字典项的，详情页显示 `ⓘ 该字典项已停用` 警告标签
- 字典项重命名记录审计日志（field: name, old_value, new_value）

#### 5.2.11 文档预览技术架构（v0.35 新增）

**核心目标**：为不同格式的 DOC 知识对象提供安全、高性能、可离线的预览能力。

**技术选型**：

| 格式 | 技术 | 说明 |
|------|------|------|
| PDF | **PDF.js**（Mozilla 开源）| 前端直接渲染，零后端依赖，离线可用 |
| DOCX / DOC / XLSX / PPTX | **后端 LibreOffice headless 转 PDF** + PDF.js | 转换结果缓存 30 天 |
| TXT / MD | 前端直接展示（等宽字体）| 无需后端 |
| PNG / JPG / JPEG / GIF | 前端 `<img>` 直接展示 | 无需后端 |
| 其他（ZIP / MP4 等）| 不支持预览，提示下载 | - |

**架构图**：

```
┌─────────┐  GET    ┌──────────────┐         ┌──────────────┐
│ KO-DOC  │ ─────→ │ 后端 API      │ ──────→ │ 对象存储      │
│ 上传     │        │ /doc/preview  │         │ MinIO/S3    │
└─────────┘        └──────────────┘         └──────────────┘
                         │                         │
                         │ ① PDF → 直链             │ ① 原始文件
                         │ ② Office → 转换服务      │ ② 转换后 PDF
                         │ ③ TXT/MD → 直链         │    （缓存 30 天）
                         │ ④ 图片 → 直链            │
                         │ ⑤ 其他 → UNSUPPORTED    │
                         ↓                         
                   ┌──────────────┐
                   │ 前端 PDF.js  │  ← 短期签名 token URL
                   │ 渲染         │     （30 分钟过期）
                   └──────────────┘
```

**接口契约**：

```yaml
GET /api/doc/{ko_id}/preview:
  desc: 文档预览入口（后端根据格式返回对应数据）
  resp:
    code: 0
    data:
      ko_id: KO-DOC-0001
      file_name: 辽港伐谋项目需求说明书 V3.1.pdf
      format: 'PDF' | 'DOCX' | 'DOC' | 'TXT' | 'XLSX' | 'PPTX' | 'PNG' | ...
      preview_strategy: 'PDFJS_DIRECT' | 'PDFJS_CONVERTED' | 'TEXT_DIRECT' | 'IMAGE_DIRECT' | 'UNSUPPORTED'
      preview_url: str  # 携带短期 token 的预览 URL（30 分钟过期）
      conversion_status: 'READY' | 'PROCESSING' | 'FAILED'  # 仅 PDFJS_CONVERTED 有
      conversion_eta: int  # 预计转换完成时间（秒）
      fallback_download_url: str
    err:
      '40401': 文档不存在
      '40301': 无访问权限（项目隔离）

GET /api/doc/{ko_id}/file:
  desc: 预览文件下载（带 token 鉴权）
  query:
    token: str  # 短期签名 token
  resp:
    - PDF: application/pdf, Content-Disposition: inline
    - TXT: text/plain; charset=utf-8
    - 图片: image/*
    - 401: token 无效或过期
```

**安全性**：
- 预览 URL 携带短期签名 token（**30 分钟过期**）
- Token 不可重放（一次性或短时效 + 服务端校验）
- SSO 用户身份验证
- 后端记录预览访问日志（审计：DOC_PREVIEW_ACCESSED）

**NFR 新增（v0.35）**：

| 编号 | 类型 | 需求 | 衡量标准 |
|------|------|------|----------|
| NFR-24 | 性能 | 文档预览性能 | ① PDF 加载 ≤ 3s（10MB 以内）<br>② Office → PDF 转换 ≤ 30s（首次）<br>③ Office → PDF 转换 ≤ 1s（缓存命中）<br>④ 大文件显示加载进度条 | 性能测试 |
| NFR-26 | 可用性 | 文档预览可用性 | ① 转换失败重试 2 次<br>② 失败后降级为下载入口<br>③ 离线场景正常加载 PDF.js（自托管）<br>④ 跨浏览器支持（Chrome/Edge/Safari） | 故障注入测试 |
| NFR-27 | 安全 | 文档预览安全 | ① 预览 URL 短期 token（≤30 分钟）<br>② Token 不可重放<br>③ SSO 用户身份验证<br>④ 预览访问记录审计 | 渗透测试 |

**验收清单**：
- PDF 预览：点击 PDF 文档名 → 弹窗 → PDF.js 渲染（多页支持）
- DOCX 预览：点击 DOCX 文档名 → 弹窗 → 后端转换 → PDF.js 渲染
- TXT 预览：点击 TXT 文档名 → 弹窗 → 等宽字体文本
- 图片预览：点击图片文档名 → 弹窗 → `<img>` 展示
- 不支持格式：点击 → 弹窗 → "暂不支持预览" + 下载入口
- 转换中：显示进度条 + ETA
- 转换失败：重试 2 次 → 降级为下载
- 大文件（>10MB）：加载提示
- 离线：PDF.js 自托管，正常加载 PDF
- Token 过期：30 分钟后拒绝访问，要求重新打开

#### 10.5.5 引用关系口径（v0.32 新增）

"引用关系"统计 KO 之间的有向引用边，**仅统计显式引用**（不包含字符串包含关系）。

| 引用类型 | 是否计入 1,247 | 示例 |
|----------|---------------|------|
| 约束（CON）通过 `@KO-XXX` 引用参数（PAR） | ✅ | KO-CON-0004 → KO-PAR-0023 |
| 约束（CON）通过 `{{var}}` 引用数据结构（SCH）字段 | ✅ | KO-CON-0007 → KO-SCH-0003.vessel_id |
| 规则（RUL）通过 If-Then 引用其他规则 | ✅ | KO-RUL-0001 → KO-RUL-0017 |
| 数据结构（SCH）引用其他 SCH（依赖）| ✅ | KO-SCH-0003 → KO-SCH-0001 |
| 提示词模板（PRM）通过 `{{var}}` 引用 PAR | ✅ | KO-PRM-0001 §7 → KO-PAR-0001 |
| 文档（DOC）作为其他 KO 的来源（追溯）| ❌（仅在 KO 详情页展示，不计入 1,247）| KO-CON-0004 ← KO-DOC-0001 |
| 同名/相似 KO 关系 | ❌（本期不统计，二期 RoadMap 见 §5.2.7）| - |

**作用域**：「统筹模型主提示词模板 KO-PRM-0001」装配范围内所有 KO 之间的引用边 = 1,247。

**统计 SQL 参考**：
```sql
WITH prm_assembly_kos AS (
  SELECT DISTINCT ko_id FROM prm_assembly WHERE prm_id = 'KO-PRM-0001'
)
SELECT COUNT(*)
FROM ko_reference kr
WHERE kr.source_ko IN (SELECT ko_id FROM prm_assembly_kos)
  AND kr.target_ko IN (SELECT ko_id FROM prm_assembly_kos);
-- 结果：1,247
```

### 10.6 冲突与健康检测类型完整定义

| 代码 | 名称 | 描述 | 当前数量 |

#### 处置策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 覆盖（Supersede） | 以特例覆盖全局，或以全局覆盖特例 | C2作用域重叠 |

#### 10.7.7 ID 生成规则

> 本章节定义平台所有对象的唯一标识符编码格式与生成规则，确保全系统 ID 可解析、可追溯、无冲突。

**10.7.7.1 知识对象（KO）**

| 属性 | 规则 |
|------|------|
| 格式 | `KO-{TYPE}-{NNNN}` |
| 说明 | TYPE 为六类代码：CON / RUL / PAR / SCH / PRM / DOC；NNNN 为 4 位自增序号（0001 起） |
| 生成方式 | 创建新 KO 时按类型独立自增，不同类型 NNNN 互不影响 |
| 示例 | `KO-CON-0015`（约束类第 15 号）、`KO-DOC-0003`（文档类第 3 号） |
| 应用位置 | KO 列表 · KO 详情 · 提示词组装器 · 审计日志 · 冲突治理 |
| 备注 | 需与所属项目 ID 关联存储，但 KO ID 本身不包含项目信息，以支持跨项目迁移（如二期开放） |

**10.7.7.2 项目（Project）**

| 属性 | 规则 |
|------|------|
| 格式 | `PROJ-{NNNN}` |
| 说明 | NNNN 为 4 位自增序号（0001 起） |
| 生成方式 | 创建新项目时平台自动分配 |
| 示例 | `PROJ-0001`（第 1 个项目）、`PROJ-0004`（第 4 个项目） |
| 应用位置 | 项目管理 · 顶部栏项目切换器 · KO 过滤 · 成员管理 |
| 备注 | 当前原型使用 PROJ-0001~PROJ-0004 |

**10.7.7.3 提示词记录（Prompt Record）**

| 属性 | 规则 |
|------|------|
| 格式 | `PRP-{NNNN}` |
| 说明 | NNNN 为 4 位自增序号（0001 起） |
| 生成方式 | 提示词渲染完成后自动分配 |
| 示例 | `PRP-001`（第 1 条提示词记录） |
| 应用位置 | 提示词组装器 · 版本快照 · 审计日志 |
| 备注 | 区别于快照（SNP）——PRP 代表一次渲染产出的提示词文本；一个 SNR 可能关联多个 PRP（同一装配方案多次渲染） |

**10.7.7.4 提示词版本快照（Snapshot）**

| 属性 | 规则 |
|------|------|
| 格式 | `SNP-{YYYYMMDD}-{HHMM}` |
| 说明 | 提示词装配方案的时间戳快照标识：YYYYMMDD 为日期，HHMM 为分钟 |
| 生成方式 | 提示词渲染触发时自动生成（取服务器时间） |
| 示例 | `SNP-20260521-1430`（2026 年 5 月 21 日 14:30 创建的快照） |
| 应用位置 | 版本快照时间线 · 顶部栏面包屑 · 提示词组装器 |
| 备注 | 快照为不可变记录，仅追加不修改；当同一分钟内有多次快照时，追加序号 `-X`（如 `SNP-20260521-1430-2`） |

**10.7.7.5 版本号（KO Version）**

| 属性 | 规则 |
|------|------|
| 格式 | `v{MAJOR}.{MINOR}.{PATCH}` |
| 说明 | 语义化版本号：MAJOR = 架构级变更（作用域调整、公式重构等）；MINOR = 功能级增改（添加子规则、新增字段等）；PATCH = 纠正性微调（数值修正、描述更正，不改变语义） |
| 生成方式 | 编辑 KO 时由用户或系统根据变更程度选择版本号递增策略 |
| 示例 | `v3.1.0`（第 3 次 MAJOR、第 1 次 MINOR）、`v1.0.0`（初始版本） |
| 应用位置 | KO 详情 · 版本快照 · 提示词组装器来源追溯行 · 审计日志 |
| 备注 | 遵循语义化版本规范；每个 KO 维护独立的版本树 |

**10.7.7.6 角色（Role）**

| 属性 | 规则 |
|------|------|
| 格式 | `ROLE-{NNNN}` |
| 说明 | NNNN 为 4 位自增序号（0001 起） |
| 生成方式 | 创建新角色时平台自动分配 |
| 示例 | `ROLE-0001`（系统管理员） |
| 应用位置 | 权限与角色 · 用户角色分配 · 审计日志 |
| 备注 | 系统预置角色（系统管理员、算法工程师、业务专家、合规审核员）使用固定 ID：`ROLE-0001`~`ROLE-0004`；自定义角色从 `ROLE-0005` 起分配 |

**10.7.7.7 用户（User）**

| 属性 | 规则 |
|------|------|
| 格式 | `USR-{NNNN}` |
| 说明 | NNNN 为 4 位自增序号（0001 起） |
| 生成方式 | 新用户创建时平台自动分配 |
| 示例 | `USR-0001`（第 1 位平台用户，如 FAN） |
| 应用位置 | 用户列表 · 角色分配 · 审计日志 · 成员管理 |
| 备注 | 对接 SSO 后，USR ID 为平台内部标识，与 SSO 账号（如邮箱地址）关联 |

**10.7.7.8 审计日志记录（Audit Log Entry）**

| 属性 | 规则 |
|------|------|
| 格式 | `AUDIT-{YYYYMMDD}-{NNNNNN}` |
| 说明 | YYYYMMDD 为操作日期，NNNNNN 为当日 6 位自增序号（000001 起） |
| 生成方式 | 记录审计日志时自动分配（按天独立自增） |
| 示例 | `AUDIT-20260605-000001`（2026-06-05 当日第 1 条审计记录） |
| 应用位置 | 审计日志列表 · 日志导出 |
| 备注 | 日志 ID 中的日期字段便于按天检索和归档；日志保留周期 ≥ 12 个月，超期自动归档 |

**10.7.7.9 字典项（Dictionary Item）**

| 属性 | 规则 |
|------|------|
| 格式 | `DICT-{CATEGORY}-{CODE}` |
| 说明 | CATEGORY 为字典类别代码，CODE 为字典项唯一标识码 |
| 生成方式 | 新增字典项时由管理员手动指定 CODE 或平台自动生成 |
| 示例 | `DICT-FORCE-Hard`（效力分级中的 Hard 硬约束项）、`DICT-AUTH-L1`（权威分级中的 L1 行业规范项） |
| 应用位置 | 字典管理 · KO 详情（标签引用） · 类型筛选 |
| 备注 | 字典值变更记录到审计日志（关联 AUDIT ID） |

**10.7.7.10 类型分组（Type Grouping）**

| 属性 | 规则 |
|------|------|
| 格式 | `GRP-{TYPE}-{NNNN}` |
| 说明 | TYPE 为所属 KO 类型代码（CON/RUL/PAR/SCH/PRM），NNNN 为 4 位自增序号（0001 起，按类型独立自增） |
| 生成方式 | 新增类型分组时平台自动分配 |
| 示例 | `GRP-CON-0001`（约束类 Hard 硬约束分组）、`GRP-PAR-0003`（参数类天气系数分组） |
| 应用位置 | 字典管理（类型分组名称 Tab） · KO 类型详情页筛选 · 提示词组装器 |
| 备注 | DOC 类型暂无子分组，因此不使用 GRP-DOC-XXXX |

**10.7.7.11 冲突检测记录（Conflict Record）**

| 属性 | 规则 |
|------|------|
| 格式 | `CT-{TYPE}-{YYYYMMDD}-{NNNN}` |
| 说明 | TYPE 为冲突类型代码（C1~C6、H1~H6），YYYYMMDD 为检测日期，NNNN 为 4 位自增序号 |
| 生成方式 | 冲突检测触发时自动生成 |
| 示例 | `CT-C1-20260605-0001`（2026-06-05 检测出的第 1 个 C1 值冲突） |
| 应用位置 | 知识治理 · KO 详情冲突内联卡片 · 审计日志 |
| 备注 | 同一冲突在未处置前保持同一 ID，处置后状态变更但 ID 不变；重新检测发现同一冲突时沿用旧 ID |

**10.7.7.12 量纲（Dimension）**

| 属性 | 规则 |
|------|------|
| 格式 | `DIM-{NNNN}` |
| 说明 | NNNN 为 4 位自增序号（0001 起） |
| 生成方式 | 新增量纲时平台自动分配 |
| 示例 | `DIM-0009`（第 9 个量纲） |
| 应用位置 | 字典管理（量纲配置 Tab） · 参数类 KO 详情 |
| 备注 | 量纲用于参数类 KO 的单位标识，初始化预制 9 个量纲 |

**10.7.7.13 ID 生成规则速查表**

| 对象类型 | ID 格式 | 自增粒度 | 示例 |
|----------|---------|----------|------|
| 知识对象 | `KO-{TYPE}-{NNNN}` | 按 TYPE 独立自增 | `KO-CON-0015` |
