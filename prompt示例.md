# 港口资源需求统筹模型算法演化 Prompt

## 0. 演化目标

当前 `init` 版本为极度简化版本。请基于本 Prompt 对代码进行完整演化，包括规则补全、约束校验、计划推测、资源统筹、效率效益优化和结果输出。

演化后的算法不得简化任何业务规则。即使输入数据缺失、字段为空、字段未定义或数据质量不稳定，也必须尽量容错处理，不能因为单个表、字段或记录异常导致整体崩溃。

## 1. 角色定义

你是世界级港口资源调度算法工程师，专门解决集装箱码头资源优化调度问题。

你的任务是在满足全部输入输出格式、硬约束、业务规则、计划推测规则、设备启发式规则和优化目标的前提下，设计并实现一个高效、稳健、可演化的港口资源需求统筹模型算法，最大化效率效益综合得分。

## 2. 核心任务

根据输入数据、输出数据、约束条件、算法规则、需求分析、计划推测规则、设备启发式规则和优化目标，完成所有纳入计算船舶的集装箱装卸任务与相关堆场作业任务。

算法需要先生成完整作业计划，再从当前实际时间的整点开始，截取之后 72 小时的小时级统计结果作为输出。

核心要求：

1. 完成所有有计划靠离泊时间且未实际离泊船舶的箱装卸任务。
2. 综合安排岸桥、场桥、拖车、正面吊、空叉等资源。
3. 统筹船舶装卸、堆场作业、移箱、提箱、外集卡收箱、班列作业等交叉任务。
4. 满足全部硬约束与业务规则。
5. 基于效率效益配置优化综合得分。
6. 对缺失或异常数据保持高容错，不因局部数据问题报错退出。

## 3. 计算范围

### 3.1 时间范围

统筹模型从当前实际时间的整点开始计算。

算法应忽略当前已经安排的设备计划，例如已有岸桥计划、场桥计划等，对所有设备从当前时间开始重新计算排程。

输出只截取当前整点之后 72 小时的数据，但算法内部应尽量生成完整计划，确保船舶任务完整性和指标计算一致性。

### 3.2 船期范围

只计算满足以下条件的船期：

1. 有计划靠泊时间 `etb_dt`。
2. 有计划离泊时间 `etu_dt`。
3. 没有实际离泊时间 `atu_dt`。

已经实际离泊的船舶不再计算。

实际靠泊但尚未实际离泊的船舶，需要根据箱作业状态筛除已完成箱子，避免重复安排作业量。

一艘船可能同时包含进口和出口任务。算法应根据申报计划中的进口航次和出口航次同时识别进口船期和出口船期。

## 4. 输入数据

输入数据来自 `输入数据.xlsx` 及相关 CSV 文件。算法必须按表名和字段含义进行解析，字段缺失时使用默认值、空集合或跳过局部规则，不得整体崩溃。

### 4.4 数据解释规则

1. 街区具体位置和场箱位：前三位为街区号，4-6 位为贝位号，7-8 位为行号，9-10 位为层号。例如 `G150110301` 表示 `G15` 街区、`011` 贝、`03` 行、`01` 层。
2. 船箱位：前三位为贝位号，4-5 位为行号，6-7 位为层号。例如 `0180282` 表示 `018` 贝、`02` 行、`82` 层。
3. 位置字段为 `EC` 或空值时，表示当前没有具体位置。
4. 可通过船舶计划的 `berth_plan_no` 在申报计划中查找船名和航次。
5. 街区基础数据中 `function` 或 `boolattr` 缺失时，该街区不可用。
6. 街区功能主要包括进口、出口、中转、机动等。街区功能和类型共同决定可堆放的集装箱类型。
7. 堆场计划和智能派位中的栏等同于街区。
8. 配载计划、航线历史作业和进口船图中包含内外贸类型。类型为空的箱不区分内外贸。
9. 内贸箱不能进入外贸街区，可进入内贸街区或未标注外贸的街区；外贸箱同理。
10. 配载计划箱和清洁清单箱可按箱号对应。
11. 箱作业状态中的箱子均视为已完成箱子，不再安排设备作业。
12. 大区编号：`C` 区为一区，`P` 区为二区，`D` 区为三区。
13. 维保计划为当天级别，当天维保设备不可用。

## 6. 硬约束

任何硬约束违反都应尽量通过重新调度、降级分配或局部修复解决。确实无法满足时，仍需输出可解释结果，并在内部约束检查中记录原因。

### 6.1 资源上限约束

任意时刻，分配给所有作业的设备数量不得超过可出勤设备数量。

公式：

```text
码头设备可出勤数 = 码头设备实际拥有数 - 码头设备不可出勤数
码头任意时刻使用设备数 <= 码头设备可出勤数
```

不可出勤数根据维保计划计算。

### 6.2 岸桥开头量约束

船舶可分配岸桥数量不得超过岸桥开头量上限。

岸桥安全要求：两个岸桥同时工作时，至少隔开 2 个小贝。

公式：

```text
岸桥数量上限 = floor((申报计划船舶贝位数 + 2) / 3)
```

### 6.3 时间窗口约束

每条船的全部装卸作业必须安排在计划靠泊到计划离泊窗口内。

```text
计划靠泊时间 <= 第一个箱任务开始时间
最后一个箱任务完成时间 <= 计划离泊时间
```

### 6.4 即到即靠约束

时间一前一后靠泊且桩位有交叉的两艘船，前一艘船预计离泊时间不得晚于后一艘船计划靠泊时间要求。

```text
预计离泊时间 = 预计完船时间 + 离泊前工作耗时 1.34h
预计离泊时间 <= 后船计划靠泊时间 - 2h
```

说明：`2h` 为船舶离泊工作耗时，该时段岸桥无作业安排，可调度给其他可作业船舶；若无可作业船舶则闲置。

若效率效益配置中即到即靠为强制要求，则对应单船必须满足；否则全局结果应尽量达到约 80% 即到即靠满足率。

### 6.5 班期服务约束

干线船和支线船有不同的在泊时长上限和单船效率下限。

```text
预计在泊时长 = 预计离泊时间 - 计划靠泊时间
单船效率 = 72小时内完成箱数量 / 72小时内实际作业时长
实际作业时长 = 72小时内最后一个箱任务完成时间 - 72小时内第一个箱任务开始时间
```

干线船：

```text
预计在泊时长 <= 干线船在泊时长上限
单船效率 >= 干线船单船效率下限
```

支线船：

```text
预计在泊时长 <= 支线船在泊时长上限
单船效率 >= 支线船单船效率下限
```

口径说明：单船指标汇总中的 `single_vessel_efficiency` 使用实际作业时长口径；效率得分中的干线/支线平均效率按第 10.1 节的评分口径计算。

### 6.6 设备移动时间约束

场桥、正面吊、空叉等设备跨街区转场时，必须留出足够移动时间。

```text
下一个任务开始时间 - 上一个任务结束时间 >= 转场预计最短耗时
转场预计最短耗时 = 街区距离 / 设备速度
```

输出场桥作业计划时，应反映设备转场后的街区归属和作业时间。

### 6.7 设备冲突约束

同一时刻：

1. 岸桥不能同时作业多艘船。
2. 场桥不能同时作业多个街区。
3. 正面吊和空叉不能同时作业多个街区。
4. 任一设备不能同时承担多项互斥任务。

### 6.8 设备工作范围约束

任一设备不得超出其工作能力范围：

1. 岸桥：受可作业大区、泊位编号、米位范围约束。
2. 场桥：受可作业街区约束。
3. 拖车：无人集卡作业 D 区，有人集卡可作业全场。
4. 空叉：受可作业街区约束。
5. 正面吊：受可作业街区约束。

### 6.9 作业暂停约束

根据作业参数执行暂停：

1. 停止类型为全船：该时间段内所有泊位、所有船舶进出口作业暂停，泊位不可用；与船舶进出口无关的部分街区箱作业可继续。
2. 停止类型为单船：该时间段内指定船舶进出口箱作业暂停，所在泊位不可用；其他作业可继续。
3. 停止类型为街区：该时间段内指定街区所有箱作业暂停，街区不可用；其他作业可继续。

## 7. 业务规则

### 7.1 天气影响效率规则

不同天气对各类设备作业效率有不同影响。

```text
设备作业效率 = 设备基础效率 * 天气效率影响百分比
```

天气影响时段为天气参数开始时间所在自然日的 0-24 点。

算法应从天气信息中提取关键字，与天气效率配置中的天气类型匹配。匹配成功后，对当天对应设备效率进行折减。

### 7.2 外集卡收箱规则

1. 所有外集卡收箱任务必须在截载计划时间内完成。
2. 优先按船名、航次、箱号查找箱子当前所在街区。
3. 若按精确箱号找不到街区，则按船名和航次查找该船箱子堆存街区，并选择一个可用街区。
4. 在计划时间内完成任务即可，不应在同一时间占用过多资源。

### 7.3 提箱规则

1. 提箱预约时间按单天处理。
2. 找到同一车队下所有预约提箱箱号。
3. 将箱量平摊到当天各小时。
4. 按箱号查找预约箱所在街区，并安排设备计划。

### 7.4 移箱规则

1. 为起始街区安排设备。
2. 起始街区为空或无法识别时，起始街区不计量、不安排设备。
3. 为目标街区安排设备。
4. 目标街区为 `EC` 时，算法应寻找繁忙度低的查验区作为目标街区，并安排设备。
5. 目标街区为空或无法识别时，目标街区不计量、不安排设备。
6. 移箱任务应合理平均分配，避免堆场堵塞。

### 7.5 火车班列规则

根据班列类型、火车到达时间、发车时间、箱量和街区安排统筹设备作业计划。

班列类型：

- `L`：发车。
- `U`：卸车。

### 7.6 街区拓扑和距离计算规则

C 区：

1. A-E 列街区，字母相邻则列相邻。
2. 每列街区数量不等，每行街区数字序号相等。
3. `F05` 与 `E15` 列相邻、行相同。
4. 示例：`A03` 到 `E06`，先算 `A03-E03` 距离 4，再算 `E03-E06` 距离 3，总距离 7。

P 区：

1. G-I 列街区，字母相邻则列相邻。
2. 每列街区数量不等，每行街区数字序号相等。
3. 示例：`G02` 到 `H06`，先算 `G02-H02` 距离 1，再算 `H02-H06` 距离 4，总距离 5。

D 区：

1. `K01-K10` 与 `Y01-Y06` 同列。
2. `L01-L10` 与 `X01-X06` 同列。
3. 字母相邻则列相邻，每行街区数字序号相等。
4. 由 K 到 Y 向下分布，由 L 到 X 向下分布。
5. 示例：`K03` 到 `X06`，先算 `K03-L03` 距离 1，再算 `L03-X06` 距离 13，总距离 14。

### 7.7 街区繁忙度计算规则

```text
街区最大堆存量 = ((最大贝 - 起始贝) / 2) * 列数 * 层数
街区当前堆存量 = 当前箱在堆场数据中同一街区内不同箱号总数
街区繁忙度 = 街区当前堆存量 / 街区最大堆存量
```

若街区最大堆存量缺失或小于等于 0，则该街区繁忙度按高风险处理，避免优先分配。

## 8. 船期任务推导规则

### 8.1 进口船期

1. 有进口船图且有堆场计划：按堆场计划的街区和箱类型为船舶找到卸箱位置，并为街区和船舶箱任务安排设备计划。
2. 有进口船图、无堆场计划、有智能派位：按智能派位要求的街区合理安排集装箱，并对进口街区和箱量安排设备计划。
3. 有进口船图、无堆场计划、无智能派位：按街区类型和进口堆场计划推测规则，对进口街区和箱量安排设备计划。
4. 无进口船图、有申报且有箱数据：按申报箱量、尺寸、空重、危险品、冷藏、超限和街区类型安排进口街区和设备计划。
5. 无进口船图、有申报但无箱数据：使用该航线相关船舶历史进口箱作业数据，排除异常数据后取平均，按街区类型安排进口街区和设备计划。

### 8.2 出口船期

1. 有配载，且配载量与清洁清单相差 5% 以内：根据配载顺序和堆场计划中的出口箱位置安排船舶和街区设备计划。
2. 有配载，但配载量与清洁清单相差大于 5%：若有清洁清单，则按无配载有清洁清单规则推测。
3. 无配载、有清洁清单：根据清洁清单做配载推算。若有船舶预配，则导入预配工况，在预配基础上推测配载顺序；否则直接推测配载顺序。
4. 有申报及申报数量、无配载、无清洁清单：按申报箱数据和街区类型安排出口街区和船舶设备计划，不做配载推算。
5. 清洁清单没有场箱位字段时，依据当前箱在堆场数据按箱号获取箱当前位置。
6. 有申报但无申报箱数据、无配载、无清洁清单：使用该航线相关船舶历史出口箱作业数据，按历史箱作业数据和街区类型安排设备计划，不做配载推算。

## 9. 计划类数据推测

### 9.1 进口堆场计划推测

当进口船期未配置堆场计划和智能派位时，算法需要为每个进口箱推测卸船后的堆存街区，并输出推测原因。

泊位到普通箱街区优先映射：

1. C 区：
   - D03#、D04# 对应 B01-B08。
   - D05# 对应 C01-C08。
   - D06# 对应 D01-D08。
   - D07# 对应 E01-E08。
   - D09#、D10# 无固定对应街区，在 C 区内按箱选择，因靠近 E 列，按就近原则可从 E 到 A 选择。
2. P 区：
   - D11#、D12#、D13# 对应 G01-G18。
   - D14# 对应 H01-H17。
   - 15# 对应 I01-I17。
3. D 区：
   - D17# 对应 K01-K08。
   - D18# 对应 L01-L08。

箱型和街区匹配规则：

1. 危险品优先安排到危险品街区。
2. 冷藏箱优先安排到冷藏街区。
3. 空箱优先安排到空箱街区。
4. 超限箱安排到可堆存超限箱的街区。
5. 普通箱、重箱、混堆箱按泊位就近街区和街区功能选择。
6. 内外贸必须匹配，类型为空的箱可不区分内外贸。

综合评分因素：

1. 街区功能和箱型匹配。
2. 内外贸匹配。
3. 靠泊位置和街区距离近。
4. 街区空闲位数多、繁忙度低。
5. 优先选择电场桥可作业街区。
6. 优先安排卸船街区，其次装卸混合街区，最后才安排包含收提箱作业的街区。
7. 一条街区最多安排该船 3 个岸桥作业量的进口箱。
8. 若船舶只开 3 个岸桥，且街区容量足够，则尽量将所有箱安排至一个街区。
9. 避免同船或靠离泊时间重叠的不同船舶不同贝位箱子集中到同一街区。
10. 避免多艘进口船分配到同一街区。
11. 若无空场，可选定合并堆场使用。

### 9.2 配载计划推测

当出口船期未进行配载时，系统需要根据清洁清单和船舶预配情况推测配载计划。

排序规则：

1. 重箱优先作业，空箱后作业。
2. 相同目的港箱子尽可能安排在一起。
3. 有预配数据时，根据预配数据推测装船顺序。
4. 同街区箱子尽可能连续排序。
5. 同街区不同贝位箱子按大贝到小贝排序，可根据街区行驶方向调整。
6. 堆场同贝位箱子按从上到下顺序作业。

## 10. 优化目标和评分

算法应根据效率效益配置调整设备计划，最大化效率效益综合得分。

当效率权重提高时，算法可适当提高设备使用率、缩短船舶作业时间、提高船舶效率。

当效益权重提高时，算法应降低资源投入成本，同时仍满足硬约束。

### 10.1 效率得分

效率目标以岸桥效率提升和船舶作业时间减少为核心，通过合理增加岸桥、拖车、场桥等资源提升装卸效率。

效率细分权重：

```text
Wse1 = 干线单船平均效率权重
Wse2 = 支线单船平均效率权重
Wab  = 即到即靠率权重
Wfc  = 班期服务达标率权重
Wse1 + Wse2 + Wab + Wfc = 100%
```

指标：

```text
Rse1 = 干线船72小时完成箱量 / 72，再按干线船数取平均
Rse2 = 支线船72小时完成箱量 / 72，再按支线船数取平均
Rab  = 符合即到即靠船舶数量 / 72小时内计划靠泊船舶数量
Rfc  = 预计在泊时长与效率均合格船数 / 总船数
Se   = Wse1*Rse1 + Wse2*Rse2 + Wab*Rab + Wfc*Rfc
```

评分计算前需要对不同量纲指标进行归一化。

### 10.2 效益得分

效益细分权重：

```text
Wec = 资源投入总成本权重
Wbc = 单箱成本权重
Wsc = 单船成本权重
Wec + Wbc + Wsc = 100%
```

固定成本参数：

```text
Cc  = 单岸桥平均成本(元/TEU) * 岸桥效率(TEU/h)
Cy3 = 三区单电轨道吊平均成本(元/TEU) * 效率(TEU/h)
Cy2 = 一/二区单电轮胎吊/轨道吊成本(元/TEU) * 效率(TEU/h)
Cy1 = 单油轮胎吊/轨道吊成本(元/TEU) * 效率(TEU/h)
Cx  = 单电空叉/正面吊成本(元/TEU) * 效率(TEU/h)
Cz  = 单油空叉/正面吊成本(元/TEU) * 效率(TEU/h)
Ctr = 单拖车平均成本(元/h)
R   = 单箱平均收入(元/TEU)
```

时间决策变量：

```text
Tc  = 所有投入岸桥总在线时长
Tt  = 所有投入拖车总在线时长
Ty3 = 所有投入三区电轨道吊总在线时长
Ty2 = 所有投入一/二区电场桥/轨道吊总在线时长
Ty1 = 所有投入油场桥总在线时长
Tx  = 所有投入电正面吊/空叉总在线时长
Tz  = 所有投入油正面吊/空叉总在线时长
```

成本和收益：

```text
Ce = Cc*Tc + Ctr*Tt + Cy3*Ty3 + Cy2*Ty2 + Cy1*Ty1 + Cx*Tx + Cz*Tz
Cb = Ce / 72小时完成总箱量
Cs = Cb * 72小时平均单船完成箱量
Et = R * 72小时完成总箱量
Rt = Et - Ce
Sc = Et - Wec*Ce + Wbc*Cb + Wsc*Cs
```

口径说明：原始材料中 `Ct` 同时用于拖车成本和总成本，容易混淆；本 Prompt 中统一使用 `Ctr` 表示拖车小时成本，`Ce` 表示资源投入总成本。

效益得分计算前也需要归一化。

### 10.3 综合得分

```text
总得分 = We * Se + (100% - We) * Sc
```

其中 `We` 来自效率效益配置中的效率占比。

### 10.4 参数配置使用规则

1. 效率效益配置中 `type=单船` 的船舶，调度时必须单独考虑个性化效率效益权重。
2. `type=全局` 的船舶统一使用全局权重参数。
3. 当效率效益配置没有数据时，算法需要自动搜索调度参数，使综合得分最高，并在输出中说明算法给定的参数。
4. 当全局配置任一关键参数缺失时，不采用全局配置全部参数，改由算法自行寻优，并输出算法参数。
5. 使用全局参数的船舶只需满足全部硬约束。
6. 使用单船参数的船舶除满足全部硬约束外，还需满足单船效率下限、在泊时长上限、即到即靠等个性化要求。
7. 单船参数中某项为空时，该项不作为硬性个性化要求。

## 11. 总体调度流程

算法应按以下流程实现：

1. 加载所有输入表，进行字段标准化、时间解析、空值处理和异常容错。
2. 识别当前 72 小时输出窗口。
3. 根据维保计划、作业参数、天气参数计算可用设备和动态效率。
4. 识别纳入计算的船舶，筛除已实际离泊船舶。
5. 根据箱作业状态筛除已完成箱子。
6. 分别生成进口船期任务、出口船期任务。
7. 对数据不完整的进口任务执行进口堆场计划推测。
8. 对数据不完整的出口任务执行配载计划推测。
9. 汇总船舶装卸、堆场、移箱、提箱、外集卡收箱、班列等所有任务，形成按时间、街区、船舶聚合的箱任务需求。
10. 生成可行的基线资源计划，优先满足硬约束。
11. 使用元启发式算法或局部搜索框架优化设备分配，评估函数为效率效益综合得分。
12. 对冲突、时间窗口、设备移动、资源上限进行迭代修复。
13. 生成完整计划和 72 小时截取输出。
14. 计算作业指标、区域资源需求和效益分析。
15. 输出所有要求表格，并保留参数使用范围和推测原因。

## 12. 设备统筹启发式规则

设备作业计划需采用元启发式算法外框架，例如遗传算法、蚁群算法、模拟退火、禁忌搜索或局部搜索。

评估函数为效率效益综合得分。以下规则作为内部启发式和局部修复规则。

### 12.1 岸桥统筹

使用数据：

- 船舶计划
- 申报计划
- 清洁清单
- 配载计划
- 船舶预配
- 进口船图
- 航线历史作业数据
- 岸桥作业计划
- 维保计划
- 岸桥基础数据
- 配载计划推测结果

规则：

1. 根据维保计划和岸桥基础数据控制可出勤岸桥上限。
2. 根据岸桥开头量约束计算单船可作业岸桥数量上限。
3. 对相关出口船舶进行配载计划推测。
4. 岸桥初始位置由岸桥作业计划中岸桥当前服务船舶反查船舶计划的泊位和米位得到。
5. 岸桥作业计划只用于识别初始位置和当前服务船舶，不采用其中的计划时间作为排程结果。
6. 根据配载计划或配载推测，在计划靠离泊时间内为船舶安排岸桥。
7. 尽量减少 72 小时内单个岸桥服务位置变化。
8. 需要转移支援时，必须考虑设备工作范围和综合转移距离最小。
9. 根据效率效益参数均衡岸桥分配。

### 12.2 场桥统筹

使用数据：

- 船舶计划
- 申报计划
- 清洁清单
- 进口船图
- 航线历史作业
- 配载计划
- 堆场计划
- 移箱计划
- 提箱预约
- 外集卡收箱截载计划
- 维保计划
- 场桥基础数据
- 街区基础数据
- 进口堆场计划推测结果
- 配载计划推测结果

规则：

1. 根据维保计划控制可作业场桥上限。
2. 轨道吊和轮胎吊转场时考虑无效转场时间。
3. 尽量减少 72 小时内单个设备服务街区变化次数。
4. 需要跨街区支援时，考虑作业范围和综合转场距离最小。
5. 有初始位置时，从初始位置开始计算转场时间；无初始位置时，选择较优街区作为初始工作街区。
6. 综合堆场计划、进口堆场推测、配载计划、外集卡收箱、提箱预约、移箱计划，计算每小时每街区箱任务量。
7. 当存在外集卡收箱截载计划但不存在预约计划时，按申报箱数量和收箱截载时间均衡分配箱量。
8. 按场桥基础数据中可作业街区和设备效率分配设备。
9. 根据效率效益参数均衡场桥分配。

### 12.3 拖车统筹

使用数据：

- 岸桥设备统筹结果
- 拖车基础配置
- 维保计划

规则：

1. 根据维保计划、拖车数和班次人数上限控制可分配拖车上限。
2. 根据船舶计划靠离泊时间、岸桥作业效率配置拖车。
3. 确保岸桥作业时间内船舶作业能够完成。
4. 每个岸桥平均配置 4 个拖车，最多 6-7 个。
5. C 区一般每岸桥 3 个拖车。
6. P 区和 D 区一般每岸桥 4 个拖车。
7. 干线船每岸桥 5-7 个拖车。

### 12.4 正面吊统筹

使用数据：

- 船舶计划
- 申报计划
- 清洁清单
- 配载计划
- 堆场计划
- 移箱计划
- 提箱预约
- 外集卡收箱截载计划
- 维保计划
- 场桥基础数据中的正面吊类型
- 街区基础数据
- 进口堆场计划推测结果
- 配载计划推测结果

规则：

1. 根据堆场计划、进口堆场推测、配载计划、外集卡收箱、提箱预约、移箱计划，计算正面吊可作业街区范围内每小时箱任务量。
2. 按场桥基础数据中的正面吊配置分配可作业正面吊。
3. 转场时考虑无效转场时间。
4. 尽量减少 72 小时内单个正面吊服务街区变化次数。
5. 有初始位置时从初始位置计算转场；无初始位置时选择较优街区开始。

### 12.5 空叉统筹

使用数据：

- 船舶计划
- 申报计划
- 清洁清单
- 配载计划
- 堆场计划
- 移箱计划
- 提箱预约

- 维保计划
- 场桥基础数据中的空叉类型
- 街区基础数据
- 进口堆场计划推测结果
- 配载计划推测结果

规则：

1. 根据堆场计划、进口堆场推测、配载计划、外集卡收箱、提箱预约、移箱计划，计算空叉可作业街区范围内每小时箱任务量。
2. 考虑班次 12 人上限，一个班次最多 12 辆空叉同时工作。
3. 按场桥基础数据中的空叉配置分配可作业空叉。
4. 转场时考虑无效转场时间。
5. 尽量减少 72 小时内单个空叉服务街区变化次数。
6. 有初始位置时从初始位置计算转场；无初始位置时选择较优街区开始。

## 13. 容错和实现要求

1. 所有输入表都可能缺失，算法必须使用空表兜底。
2. 所有字段都可能缺失，算法必须使用安全读取方式。
3. 时间字段可能为空、格式异常或跨天，必须容错解析。
4. 数值字段可能为空、字符串化或异常，必须安全转换。
5. 位置字段可能为 `EC`、空值或无法识别，不能导致报错。
6. 某类设备为空时，应输出空计划或 0 资源需求，而不是崩溃。
7. 某船缺少箱级数据时，应按申报数据或历史数据补齐任务量。
8. 某船连申报数据也不足时，应跳过无法推断部分，并保留可解释原因。
9. 输出文件必须字段完整、类型稳定、时间粒度统一。
10. 算法应优先生成可行解，再优化得分。
11. 若全部硬约束无法同时满足，应输出最佳可行近似方案，并记录未满足原因。
12. 不得将规则写死到单一测试数据，应根据输入动态识别船舶、街区、设备、时间和参数。

## 14. 最终交付要求

实现后的算法必须：

1. 完整读取输入数据。
2. 完整应用本 Prompt 中的硬约束、业务规则、计划推测规则和设备启发式规则。
3. 生成所有要求的输出表。
4. 输出当前整点之后 72 小时小时级计划。
5. 计算效率、效益和综合得分。
6. 对缺失或异常数据保持稳定运行。
7. 在输出中体现参数使用范围、算法自定参数、进口堆场推测原因和效益分析。

## 15. 评估器要求

评估器用于验证算法输出是否符合格式要求、是否满足硬约束，并根据输出结果计算效率效益得分和综合评估得分。算法实现应保证输出结果能够被评估器稳定读取和校验。

### 15.1 评估器核心功能

评估器至少需要完成以下功能：

1. 验证算法输出格式是否正确。
2. 验证算法输出结果是否满足硬约束。
3. 根据规则测算效率效益总得分。
4. 返回统一评估结果，包括有效性、综合得分和错误信息。

### 15.2 必需输出文件

算法输出目录中必须包含以下文件：

| 文件名 | 内容 |
| --- | --- |
| `每小时资源需求汇总表.csv` | 每小时船舶资源需求汇总，对应船舶资源需求汇总表 |
| `每台设备逐小时作业计划.csv` | 每台设备逐小时作业计划，需合并岸桥、场桥、拖车、正面吊、空叉等设备作业计划 |
| `作业指标汇总.csv` | 船舶作业指标汇总 |
| `效益分析.csv` | 船舶或全局效益分析 |
| `参数与权重使用记录.csv` | 全局、单船或算法自定的效率效益参数和权重使用记录 |
| `每个区域的资源需求汇总表.csv` | 区域资源需求汇总 |

若算法内部仍生成细分输出，例如岸桥作业计划、场桥作业计划、拖车作业计划、正面吊计划、空叉计划，也必须保证这些细分计划能够汇总到 `每台设备逐小时作业计划.csv`。

### 15.3 输出文件基础校验

评估器应对必需输出文件执行基础校验：

1. 文件必须存在。
2. 文件必须可被 UTF-8 CSV 读取。
3. 文件不能为空。
4. 文件字段必须覆盖对应输出表要求。
5. 时间字段粒度必须为 1 小时。
6. 同一设备、同一时间段不得出现互斥重复作业记录。

如果缺少任一必需文件、文件为空或无法读取，评估器应判定输出格式错误。

### 15.4 硬约束校验

评估器应根据输入数据和算法输出，对第 6 节硬约束进行校验：

1. 资源上限约束。
2. 岸桥开头量约束。
3. 时间窗口约束。
4. 即到即靠约束。
5. 班期服务约束。
6. 设备移动时间约束。
7. 设备冲突约束。
8. 设备工作范围约束。
9. 作业暂停约束。

若任一硬约束违反，评估器应记录具体违规信息。是否将该方案判定为无效，应根据评估器配置决定；默认硬约束违反时 `validity=0`。

### 15.5 效率效益得分计算

评估器应基于算法输出计算效率效益得分，至少包括：

1. 平均单船效率。
2. 干线单船平均效率。
3. 支线单船平均效率。
4. 即到即靠率。
5. 班期服务达标率。
6. 资源投入总成本。
7. 单箱成本。
8. 单船成本。
9. 72 小时总收入。
10. 效率得分、效益得分和效率效益总得分。

效率效益得分应遵循第 10 节的公式和参数配置。若算法输出中已提供得分，评估器仍应能够根据输出计划独立复算关键指标，用于一致性校验。

### 15.6 综合评估得分

暂不将算法复杂度或运行耗时纳入综合评分。

综合评估得分直接采用效率效益总得分：

```text
combined_score = efficiency_benefit_score
```

评估器可以记录算法运行耗时用于日志和排查，但该耗时不参与 `combined_score` 计算。

### 15.7 评估器返回格式

评估器应返回结构化结果，至少包含以下字段：

| 字段 | 含义 |
| --- | --- |
| `validity` | 是否有效，1 表示输出格式正确且满足硬约束，0 表示无效 |
| `combined_score` | 综合评估得分 |
| `cost_time` | 算法运行耗时，单位秒，可选记录，不参与评分 |
| `error_info` | 错误信息或硬约束违规信息 |
| `efficiency_benefit_score` | 效率效益总得分 |
| `evaluation_score` | 评估总分，可与 `combined_score` 保持一致 |

### 15.8 评估器容错要求

评估器应具备以下容错能力：

1. 当某个输出文件缺失时，应返回明确错误信息，而不是崩溃。
2. 当某个字段缺失时，应指出缺失字段和所属文件。
3. 当时间字段无法解析时，应指出具体文件、字段和样例值。
4. 当硬约束校验过程异常时，应记录异常信息并返回 `validity=0`。
5. 当输入数据缺失或为空时，应使用空表兜底，并尽量完成可校验部分。
